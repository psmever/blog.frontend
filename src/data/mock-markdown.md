## 오늘 정리할 내용

- 프로젝트 배경과 목표
- 빠르게 릴리즈한 이유
- 다음 스프린트 계획

> 기록은 곧 자산이 됩니다. 작게 시작해도 괜찮아요.

### 체크리스트

1. API 연결 점검
2. 글 작성 화면 확인
3. 배포 파이프라인 구축

## 긴 본문 테스트

아래는 스크롤을 충분히 유도하기 위한 더미 문단입니다. 반복된 섹션이지만 제목과 리스트, 코드, 인용구를 섞어 구성했습니다.

### 섹션 A

프로젝트 초반에는 작은 범위로 시작하는 것이 중요합니다. 우선순위를 명확히 정하고, 가장 영향이 큰 작업부터 처리합니다.

- 작은 기능 단위로 쪼개기
- QA 가능한 스코프로 정의
- 작업 기록을 남기기

> 작은 개선도 누적되면 큰 차이를 만든다.

```ts
type Checklist = {
    title: string;
    done: boolean;
};

const items: Checklist[] = [
    { title: "문서 정리", done: true },
    { title: "UI 튜닝", done: false },
];
```

### 섹션 B

성능 이슈는 대부분 측정 지점이 없어서 놓칩니다. 지표를 수집하고, 병목을 확인한 뒤 필요한 최적화를 적용합니다.

1. 사용자 흐름 정의
2. 지표 수집 (TTFB, CLS, LCP)
3. 개선 전/후 비교

### 섹션 C

개발 과정의 기록은 다음 작업의 기준점이 됩니다. 팀원이 늘어날수록 컨텍스트 공유 비용이 커지므로, 가능한 한 문서화를 권장합니다.

### 섹션 D

작성 도구가 불편하면 기록이 줄어듭니다. 입력 도구의 만족도를 높이면 작성량이 자연스럽게 늘어납니다.
